# 云构建部署功能说明

## 功能概述

云构建部署现已改为页内二级菜单模式，当前提供两个子菜单：

- 我的世界Java核心开服包
- 我的世界整合包构建

该功能通过 `https://tools.xiaozhuhouses.asia/` 提供的开放接口完成目录查询、构建任务提交、任务轮询和产物下载，适合在本地环境不完整、Java 依赖不稳定或希望快速落地服务端文件时使用。

## 页面结构

进入“游戏部署”页面后：

1. 点击一级标签“云构建部署”
2. 页面会自动展开二级菜单
3. 默认显示第一个子菜单“我的世界Java核心开服包”
4. 可切换到“我的世界整合包构建”提交整合包服务端构建任务

当前旧版云构建中的 Modrinth 分支、缓存列表和旧统计面板已经移除，不影响独立主标签“Minecraft整合包部署”。

## 使用流程

### 1. 选择核心参数

在左侧“核心参数”面板中依次配置：

- 核心类型
- 核心版本
- MC版本
- User-Agent（可选）

说明：

- `MC版本` 默认会跟随所选核心版本自动填充
- 如果上游接口需要不同的 `mcVersion`，可以手动修改
- 切换核心类型后会重新加载该类型对应的版本列表
- `User-Agent` 输入框默认会显示面板默认 UA `GSM3-CloudBuild/1.0`
- Java 核心开服包子菜单的 UA 为独立配置，修改后会自动保存到浏览器，下次打开页面会自动恢复
- 仅在上游接口或下载链路对默认请求头有限制时再手动改写

### 2. 设置部署路径

在右侧“部署配置”面板中填写部署路径：

- 默认会优先使用面板已配置的默认游戏目录
- 选择核心版本时会自动生成一个推荐路径
- 你仍然可以手动修改为自定义目录

### 3. 开始构建部署

点击“开始构建部署”后，面板会按以下顺序执行：

1. 查询核心目录
2. 提交组装任务
3. 每 2 秒轮询一次任务状态
4. 任务成功后自动下载 ZIP
5. 后端自动解压到目标目录
6. 自动检测启动命令

### 4. 创建实例

部署完成后可以直接点击“创建实例”：

- 实例名称默认使用 `核心类型-核心版本`
- 描述会自动带上 MC 版本信息
- 启动命令优先使用后端扫描出的 `run/start` 脚本

## 整合包构建流程

### 1. 填写整合包来源

在“我的世界整合包构建”子菜单中依次填写：

- 来源平台（当前为 `Modrinth`）
- 整合包来源
- 可选版本号
- User-Agent（可选）
- 部署路径

说明：

- `整合包来源` 支持 Modrinth 项目页、版本页、slug 或项目 ID
- `版本号` 可留空，由上游接口自动选择默认版本
- `User-Agent` 输入框默认会显示面板默认 UA `GSM3-CloudBuild/1.0`
- 整合包构建子菜单的 UA 为独立配置，修改后会自动保存到浏览器，下次打开页面会自动恢复
- 失焦时页面会尝试根据来源自动生成推荐部署路径

### 2. 提交云端构建

点击“开始整合包构建”后，面板会按以下顺序执行：

1. 提交整合包构建任务
2. 每 2 秒轮询一次任务状态
3. 任务成功后自动下载 ZIP
4. 后端自动解压到目标目录
5. 自动检测启动命令

### 3. 创建实例

部署完成后可以直接点击“创建实例”：

- 实例名称默认使用 `整合包名称-版本号`
- 描述会自动带上 MC 版本与加载器信息
- 启动命令优先使用后端扫描出的 `run/start` 脚本

## 接口设计

### 前端调用入口

位置：`client/src/utils/api.ts`

当前云构建相关方法：

```typescript
async getCloudBuildCatalog(coreType?: string)

async createCloudBuildTask(data: {
  coreType: string
  version: string
  mcVersion: string
  userAgent?: string
})

async getCloudBuildTaskStatus(requestId: string, accessToken: string, userAgent?: string)

async createCloudModpackBuildTask(data: {
  platform: string
  source: string
  version?: string
  userAgent?: string
})

async getCloudModpackBuildTaskStatus(requestId: string, accessToken: string, userAgent?: string)

async downloadAndExtractCloudBuild(data: {
  downloadUrl: string
  targetPath: string
  archiveFileName?: string
  userAgent?: string
})
```

### 面板后端路由

位置：`server/src/routes/cloudBuild.ts`

挂载前缀：

```text
/api/cloud-build
```

接口列表：

| 路由 | 方法 | 说明 |
|------|------|------|
| `/catalog` | GET | 查询核心目录，可带 `coreType` |
| `/build` | POST | 提交组装任务，可选 `userAgent` |
| `/build/:requestId` | GET | 查询任务状态，需 `accessToken`，可选 `userAgent` |
| `/modpack/build` | POST | 提交整合包构建任务，可选 `userAgent` |
| `/modpack/build/:requestId` | GET | 查询整合包任务状态，需 `accessToken`，可选 `userAgent` |
| `/download` | POST | 下载并解压构建产物，可选 `userAgent` |

### 上游开放接口链路

固定服务地址：

```text
https://tools.xiaozhuhouses.asia/
```

调用流程：

```text
GET  /api/tools/minecraft-java-core-assembler/catalog
POST /api/open/tools/minecraft-java-core-assembler/execute
GET  /api/open/tools/minecraft-java-core-assembler/tasks/:requestId

POST /api/open/tools/minecraft-modpack-assembler/execute
GET  /api/open/tools/minecraft-modpack-assembler/tasks/:requestId
```

其中任务状态查询必须携带：

```text
X-Task-Access-Token
```

如果页面填写了 `User-Agent`，面板会将其透传到任务提交、状态轮询以及构建产物下载阶段；留空时使用服务端默认值 `GSM3-CloudBuild/1.0`。

## 后端处理细节

### 下载与解压

- 后端优先使用任务结果中的 `downloadUrl`
- 如果 `downloadUrl` 为相对路径，会自动补全到 `https://tools.xiaozhuhouses.asia/`
- 如果前端填写了 `User-Agent`，下载阶段会沿用同一个 UA 请求构建产物
- ZIP 临时文件保存在 `server/data/cloud-build/temp`
- 解压完成后会立即清理临时压缩包
- 如果压缩包内部只有一层包装目录，后端会自动展开到目标目录，避免额外多套一层文件夹

### 启动命令检测

检测规则如下：

Windows：

1. `run.bat`
2. `start.bat`

Linux / Unix：

1. `run.sh`
2. `start.sh`

## 常见问题

### 1. 获取不到核心类型

建议检查：

- 面板是否能访问 `https://tools.xiaozhuhouses.asia/`
- 是否可直接打开云构建开放接口页面确认服务可达

### 1.1 整合包来源无法识别

建议检查：

- 是否填写了有效的 Modrinth 项目链接、版本链接、slug 或项目 ID
- 目标版本是否真实存在
- 面板是否能访问 `https://tools.xiaozhuhouses.asia/`

### 2. 任务一直未完成

当前实现为轮询模式，默认每 2 秒检查一次。若长时间未结束：

- 观察日志面板中的状态信息
- 检查上游任务是否返回了排队信息
- 确认目标核心版本与 MC 版本参数是否正确

### 3. 下载或解压失败

建议检查：

- 部署路径是否可写
- 磁盘空间是否足够
- 资源站 `https://rs.xiaozhuhouses.asia/modules/minecraft` 是否可访问
